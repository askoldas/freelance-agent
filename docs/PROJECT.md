# Product Definition

## Product name

Working name: **Freelance Agent**.

## Problem

Freelance opportunities are fragmented across project boards, communities, agencies, company sites, alerts, and direct referrals. Manually checking these sources is repetitive, while generic matching often overrates weak opportunities and produces impersonal proposals.

The product should actively search for suitable projects, reduce discovery and evaluation time, and keep the owner in control of every response.

## Product objective

Build a personal project-acquisition system that:

- automatically searches configured sources for available projects;
- normalizes and deduplicates opportunities from different sources;
- understands the requested business outcome, not only keywords;
- evaluates whether the owner can responsibly deliver the project;
- identifies direct, adjacent, learnable, and risky requirements;
- recommends plausible technical approaches;
- notifies the owner only about opportunities that deserve attention;
- creates honest, tailored proposals on request;
- tracks applications and follow-ups.

Manual opportunity submission is a fallback and diagnostic feature, not the core product.

## Primary flow

```text
Scheduled search
→ collect available projects
→ normalize and remove duplicates
→ apply deterministic prefilters
→ extract project facts
→ evaluate fit and risk
→ store qualified opportunities
→ notify through Telegram
→ prepare proposal when requested
```

## Source strategy

The source layer must be modular.

### Initial sources

- public web search through Tavily;
- public project boards, agency pages, company pages, communities, and other pages discoverable through search;
- configurable search queries and domain filters.

### Later sources

- project-alert emails generated by services where the user has an account;
- official APIs and feeds where available;
- direct public-source adapters;
- a user-triggered way to send a listing from the browser to the agent;
- other supported integrations for restricted sources.

The system must not depend on a single marketplace or search provider.

## Target opportunity types

### Web products

- business and multilingual websites
- web applications
- customer portals and account areas
- directories, marketplaces, calculators, and membership products
- dashboards and internal tools
- booking, payment, and content-driven systems

### Existing systems

- finishing incomplete projects
- repairing authentication, deployment, integrations, or data flow
- extending React/Next.js or adjacent applications
- refactoring maintainability problems
- localization and multilingual implementation
- performance and reliability improvements

### CMS work

- Payload CMS
- WordPress
- headless CMS platforms
- custom administration areas
- content migrations and structured editorial workflows

### Automation and AI

- n8n or custom workflow automation
- lead research and qualification
- email and follow-up workflows
- Telegram bots
- CRM, spreadsheet, Gmail, Outlook, and API integrations
- document processing and report generation
- internal knowledge assistants
- AI agents and human-in-the-loop workflows
- non-AI business-process automation

### Consulting

- MVP planning
- architecture and stack selection
- implementation estimates
- automation feasibility
- migration planning
- review of an existing codebase or workflow

## Search philosophy

Search by business problem, desired outcome, and project-request language as well as by technology.

Examples:

- build or finish a customer portal
- automate a sales or research workflow
- replace a manual spreadsheet process
- connect a CRM, email, and reporting system
- build a multilingual service platform
- create an admin panel or CMS-driven site
- repair or complete an existing application

Technology-specific searches remain useful, but they are only one search track.

## Fit levels

### Direct fit

The owner already has strong, relevant delivery experience.

### Adjacent fit

The exact product or tool differs, but the underlying architecture and work are familiar.

### Learnable fit

Some components are new, but the scope is controlled and learning can occur without endangering delivery.

### Risky fit

The project depends on deep specialist knowledge or constraints that cannot responsibly be acquired during delivery.

## Evaluation output

Each evaluation should provide:

- total score from 0 to 100;
- primary and secondary categories;
- fit level;
- concise project summary;
- strong-match reasons;
- missing or uncertain requirements;
- delivery risks;
- learning effort;
- credibility and budget observations;
- one or more possible solution approaches;
- suggested questions for the client;
- recommendation: `priority`, `apply`, `review`, `skip`, or `reject`.

## Notification behavior

The agent should not send every raw result to Telegram.

It should:

- silently discard obvious duplicates and irrelevant results;
- keep lower-confidence results available for diagnostics when useful;
- notify immediately about strong matches;
- optionally send a compact digest for medium-confidence matches;
- include the source link, score, fit level, key reasons, risks, and actions.

## Proposal rules

A generated proposal must:

- begin with the actual client problem;
- be tailored to the listing;
- mention only verified skills and experience;
- explain a practical next step or likely approach;
- avoid excessive stack dumping;
- include one useful clarification question;
- remain concise unless the opportunity requires detail;
- remain reviewable by the owner.

## Initial non-goals

- fully autonomous applications
- automatic pricing commitments
- a multi-user SaaS product
- a complex analytics dashboard
- depending on manual project submission as the main workflow

## Success criteria for the first useful milestone

The milestone is successful when the system can run a scheduled search, discover real available projects, remove duplicates, evaluate them, store the useful opportunities, and send actionable Telegram notifications without the owner manually supplying each listing.
